## Protocol: Redundant status field in response messages

**Issue 7: Status field carries no information** ✅ RESOLVED (Story 20.4)

The three server→client response message types each contain a status field that is always fixed to the same value:

| Message type | Status field | Always |
|-------------|-------------|--------|
| `BTC_CHUNK_ACK\|<session>\|<chunk>\|OK\|...` | `OK` | Always OK — being an ACK already implies success |
| `BTC_ACK\|<session>\|SUCCESS\|TXID:<txid>` | `SUCCESS` | Always SUCCESS — the ACK prefix already means success |
| `BTC_NACK\|<session>\|ERROR\|<detail>` | `ERROR` | Always ERROR — the NACK prefix already means error |

The message type prefix already communicates the status. The status field is redundant and:
- Wastes bytes in every response (relevant for LoRa payload constraints)
- Adds unnecessary parsing complexity
- Introduces impossible invalid states (e.g. `BTC_ACK` with non-`SUCCESS` status) that parsers must guard against

**Proposed fix:** Remove the status field from all three response types. Simplified formats:

```
BTC_CHUNK_ACK|<session>|<chunk>|REQUEST_CHUNK|<next>
BTC_CHUNK_ACK|<session>|<chunk>|ALL_CHUNKS_RECEIVED
BTC_ACK|<session>|TXID:<txid>
BTC_NACK|<session>|<error_detail>
```

See **Story 20.4** in tasks.txt.

## Device Scanning: Second Meshtastic device silently hidden

**Issue 9: `scan_meshtastic_devices()` only showed one of two connected devices** ✅ FIXED
- `core/meshtastic_utils.py`'s `scan_meshtastic_devices()` delegated entirely to
  `meshtastic.util.findPorts()`, which filters serial ports to a hardcoded VID
  whitelist (Adafruit `0x239a`, Espressif `0x303a`) and only falls back to
  "everything not blacklisted" when **zero** whitelisted-VID ports are found.
- With two real devices connected — one Espressif-VID (whitelisted) and one
  Seeed-VID `0x2886` (neither whitelisted nor blacklisted) — the whitelist
  branch found a hit and short-circuited, so the Seeed device never appeared
  in the GUI's device dropdown at all.
- Not related to the Story 22.2 client/sender.py refactor; a pre-existing gap
  in the upstream `meshtastic` library's heuristic.
- Fix: `scan_meshtastic_devices()` now enumerates ports directly via
  `serial.tools.list_ports.comports()`, filtering only by `meshtastic.util`'s
  (narrow) blacklist, reusing `eliminate_duplicate_port()` for the legitimate
  same-device-listed-twice case. All non-blacklisted devices now surface.
- Tests updated in `tests/test_meshtastic_utils.py` and `tests/test_btcmesh_gui.py`
  to mock `serial.tools.list_ports.comports` instead of `meshtastic.util.findPorts`.

## Device Connection: GUI got permanently stuck / first device in list unselectable

Found while manually testing the Issue 9 fix above with two real devices
connected — both bugs only surface with 2+ real devices, which the
`findPorts()` whitelist bug had made impossible to hit before now.

**Issue 10: Transient connect error left the GUI stuck forever** ✅ FIXED
- `btcmesh_gui.py`'s `_init_meshtastic()` treats `"resource temporarily
  unavailable"` / `"busy"` errors from `MeshtasticSerialTransport.connect()`
  as transient, showing "Device is initializing, please wait..." — but there
  was no retry and no timeout anywhere. The connection attempt simply ended
  there, leaving the GUI stuck on that message permanently with no recovery
  short of restarting the app.
- Reproduced by selecting one device, then quickly selecting a second device
  before the first had fully disconnected — the second device's
  `SerialInterface` open transiently failed and the GUI hung.
- Separately, `_disconnect_device()`'s background close of the previous
  transport silently swallowed any exception (`except Exception: pass`),
  so a failed disconnect gave no visible indication that a port might still
  be locked.
- Fix: `_init_meshtastic()` now retries up to `CONNECT_MAX_ATTEMPTS` (4) times
  with a `CONNECT_RETRY_DELAY_SECONDS` (1.5s) delay between attempts on a
  transient error, reporting `connection_error` if it still hasn't succeeded
  after all attempts instead of hanging indefinitely. `_disconnect_device()`
  now logs a warning message (via the existing `'log'` result type) instead
  of silently discarding disconnect failures.
- Tests added in `tests/test_btcmesh_gui.py`
  (`TestDeviceConnectionRetryAndSelectionFix`):
  `test_init_meshtastic_retries_transient_error_then_succeeds`,
  `test_init_meshtastic_gives_up_after_max_attempts`.

**Issue 11: First device in a multi-device list couldn't be selected** ✅ FIXED
- When multiple devices are found, `_handle_result()`'s `'devices_found'`
  handling set `device_spinner.text = devices[0]` as a display-only
  placeholder (explicitly not auto-connecting, to let the user choose).
- Kivy's `Spinner.text` (a `StringProperty`) only dispatches its `on_text`
  event when the value actually *changes*. Since the placeholder was set to
  the exact same string as the first dropdown option, selecting that first
  device again from the dropdown didn't change `.text` at all — so
  `on_device_selected` never fired, and nothing happened. Selecting a
  *different* device worked fine (the text genuinely changed), and then
  going back to the first device also worked (now a real change from the
  second device's text).
- Fix: use a new placeholder constant `SELECT_DEVICE_TEXT` ("Select a device
  to connect...") that never collides with any real device path, so any
  device selection — including the first one — always registers as a text
  change.
- Tests added: `test_devices_found_multiple_sets_placeholder_not_first_device`,
  `test_on_device_selected_ignores_placeholder_text`.

## Transport: Failed handshake leaked the serial port lock

**Issue 12: A connect() failure during the device handshake leaked the open
serial port and reader thread** ✅ FIXED
- Found while investigating a real `"Timed out waiting for connection
  completion"` error from `meshtastic.serial_interface.SerialInterface()`
  (an upstream 30s wait for the device's config handshake to finish).
- `SerialInterface(devPath=...)` (default `connectNow=True`) performs the
  entire handshake *inside its own constructor* — opens the serial port,
  starts a background reader thread, sends `want_config_id`, then blocks on
  `_waitConnected()`. If that wait times out, the exception is raised from
  inside `__init__`, so `iface = SerialInterface(...)` in
  `transport/meshtastic_serial.py`'s `connect()` never completes its
  assignment. That left no reference to the already-opened port/reader
  thread to call `.close()` on — leaking an exclusive OS-level lock on the
  serial port for the rest of the process's lifetime (confirmed via `lsof`:
  the port stayed locked by the running process even after the connection
  attempt had already failed and raised).
- Verified live against real hardware: reproduced the exact timeout with the
  plain `meshtastic` library directly (bypassing our code entirely), proving
  the timeout itself isn't caused by btcmesh — but the leak was.
- Fix: construct with `connectNow=False` (opens the port but skips the
  handshake), then call `iface.connect()` / `iface.waitForConfig()`
  ourselves in a try/except that calls `iface.close()` on failure — so we
  always keep a reachable reference to clean up, even when the handshake
  fails partway through.
- Test added in `tests/test_meshtastic_serial_transport.py`:
  `test_connect_closes_iface_when_handshake_times_out`.
- Note: the underlying 30s handshake timeout itself is a real, separate
  device/firmware-level behavior (in this case traced to both test devices
  being stuck mid-boot from rapid repeated connect/disconnect cycles during
  testing — resolved by a physical power cycle) and isn't something this fix
  changes; this fix only prevents the *leak* when that timeout occurs.

## Known Nodes: Crash sorting nodes with lastHeard=None

**Issue 13: `get_known_nodes()` crashed when a node's lastHeard was explicitly
None** ✅ FIXED
- Found by manually testing the client GUI's "Scan for nodes" button (the
  first time this feature was exercised end-to-end against a real, long-lived
  node database of 71 nodes — earlier testing past days only exercised device
  scanning/connecting via one-off scripts, never this code path through the
  actual running GUI).
- `core/meshtastic_utils.py`'s `get_known_nodes()` extracted each node's
  timestamp via `node_data.get('lastHeard', 0)`. `dict.get(key, default)`
  only applies the default when the key is *missing* — not when it's present
  but explicitly `None`, which is how the `meshtastic` library represents a
  node it knows about (via mesh gossip/NeighborInfo) but has never actually
  received a packet from directly. The resulting `None` then crashed
  `nodes.sort(key=lambda n: n['lastHeard'], reverse=True)` with
  `TypeError: '<' not supported between instances of 'NoneType' and 'int'`
  as soon as at least one such node existed alongside nodes with real
  timestamps.
- Real, pre-existing bug unrelated to the Story 22.3/22.4 rename work or
  Issues 9-12 — confirmed against real hardware: of the device's 71 known
  nodes, 3-4 had `lastHeard: None`.
- Fix: `(node_data.get('lastHeard') or 0)` instead of
  `node_data.get('lastHeard', 0)`, normalizing both "key missing" and
  "key present but None" to `0` uniformly.
- Test added in `tests/test_meshtastic_utils.py`:
  `test_sorts_without_crashing_when_lastheard_is_none`.

## Known Nodes: lastHeard doesn't reflect recent direct-message traffic

**Issue 14: Known-nodes sort order may not reflect recent BTC_TX/ACK protocol
exchanges** 🔍 SUSPECTED / OPEN — needs further investigation another day

- Observed while testing two real devices that had confirmed, successful
  bidirectional BTC_TX/ACK protocol message exchanges earlier the same day
  (verified via actual successful client/server test runs, in both role
  directions). Despite that, each device's `lastHeard` entry for the other
  was still 147-255 days stale — not refreshed by that day's traffic at all.
- Ruled out so far:
  - Not the Issue 13 crash/None-handling bug (separate, already fixed).
  - Not a hardware/antenna/range issue (physically confirmed fine).
  - Not a LoRa config mismatch (`region`, `modem_preset`, primary channel
    name and PSK all verified identical on both devices).
  - Sending a fresh direct text message live during investigation and
    reconnecting ~70s later still showed no update to the peer's `lastHeard`.
- Current best hypothesis: Meshtastic's `lastHeard` NodeDB field tracks mesh
  *presence* — updated by periodic broadcast beacons (NodeInfo, Position,
  Telemetry announcements) — rather than direct, addressed application
  messages like our BTC_TX/ACK protocol traffic. The `meshtastic` Python
  library's `_onTextReceive()` → `_receiveInfoUpdate()` does wire up a
  `lastHeard` update for received text messages, but that may only affect
  the actively-connected client's own in-memory session view, not what the
  device firmware persists into its onboard NodeDB (the authoritative source
  re-sent to any client on reconnect, which is what `get_known_nodes()`
  reads). Not yet confirmed against firmware source/documentation.
- Practical effect: `get_known_nodes()`'s sort-by-`lastHeard`-descending is
  working exactly as designed, but the underlying data may not represent
  what a user intuitively expects ("recency of any successful contact,
  including our own protocol's direct messages") — it may only represent
  recency of unrelated mesh presence beacons.
- Next steps to investigate: confirm against Meshtastic firmware
  source/docs whether direct/unicast application packets update the onboard
  NodeDB's `lastHeard` at all; if not, consider whether btcmesh should track
  "last successful protocol exchange" separately from the device's own
  NodeDB-reported `lastHeard` for display purposes.

## Reassembler: no enforcement of strict in-order chunk arrival

**Issue 15: `TransactionReassembler.add_chunk()` allows a session to be
created by (or a chunk added out of order into) any chunk_num, not just the
next expected one** 🔍 OPEN — follow-up, not part of Story 23.1

- Found while reviewing `server/receiver.py`'s (Story 23.1) test coverage
  for malformed/mismatched chunks and discussing edge cases in
  `core/reassembler.py`'s `add_chunk()`.
- Traced through the actual logic: if a brand new session's *first* seen
  chunk is, say, `2/3` (not `1/3`), `add_chunk()` happily creates a session
  with `total_chunks=3` (taken from that message) and stores chunk 2 under
  key `2` — no error, no NACK. It just waits for chunks 1 and 3 to arrive
  before reassembling (order-independent at completion time, since it
  iterates `1..total_chunks` from a dict keyed by chunk_num).
- Separately, `TransactionReceiver._on_message()`'s ACK logic naively sends
  `REQUEST_CHUNK|{chunk_num + 1}` — for a chunk `2/3` this asks for chunk 3
  next, even though chunk 1 is the one actually still missing. Inherited
  unchanged from the original `btcmesh_server.py` logic, not introduced by
  Story 23.1.
- In practice this shouldn't occur with our own `TransactionSender` client,
  which only ever sends chunks strictly in order (1, 2, 3, ...) in a
  stop-and-wait fashion, waiting for each ACK before sending the next. This
  is a latent robustness gap for any other/future client implementation
  that might send out of order, or a duplicate/retransmission race, not an
  active bug in current client/server behavior.
- Discussed and decided as a deliberate scope decision: **not** folding this
  into Story 23.1 (which only extracts `server/receiver.py`'s orchestration
  around the *existing, unchanged* `TransactionReassembler`) — this would
  touch shared production code (`core/reassembler.py`, used by the
  *current* `btcmesh_server.py` too) and its existing test suite, better
  done as its own focused change.
- Agreed design direction for whenever this is picked up: reject only the
  out-of-order chunk (NACK it) while leaving the in-progress session alive,
  so the actually-expected next chunk can still arrive and complete
  reassembly normally — rather than discarding the whole session (as a
  `total_chunks` mismatch currently does), which would kill otherwise-
  recoverable progress over one anomalous/duplicate message.

## Transport: Unreliable/inconsistent chunk delivery between two specific devices

**Issue 16: ESP32→Seeed chunk delivery intermittently fails completely, with
no consistent root cause identified** 🔍 OPEN — documented as environmental
limitation, not a code bug

- Found while pausing Story 23.2 to investigate a user report: sending a
  full transaction from the ESP32 device (as client) to the Seeed device
  (as server) consistently failed - the server's log showed **zero** chunks
  ever received, across repeated attempts, using both `btcmesh_client_cli.py`
  and `btcmesh_client_gui.py` as the sender, and both the current (old)
  `btcmesh_server_gui.py` and a direct `TransactionReceiver` test script as
  the receiver. The reverse direction (Seeed client → ESP32 server) worked
  reliably (occasional single retries, normal for LoRa).
- Investigated and ruled out, in order:
  1. **My Story 23.1 destination-filter fix** (`transport/meshtastic_serial.py`)
     — ruled out because the *old* `btcmesh_server.py` (which doesn't use
     that transport at all) also received nothing in the failing direction.
  2. **EU868 duty-cycle exhaustion on the ESP32** (both devices are
     configured for the EU_868 region, which enforces a strict transmit
     duty cycle in firmware) — ruled out because ESP32 successfully
     transmitted ACK/NACK replies back to Seeed in the working direction,
     proving its radio could still transmit.
  3. **Packet-size-dependent reliability** (full ~190-byte chunk messages
     failing while short ~35-byte ACK messages succeed on the same
     ESP32→Seeed link) — looked promising initially (matched the pattern of
     that specific test), but a direct, protocol-bypassing raw-message test
     (a plain short text message and a full-chunk-sized text message, sent
     directly via `transport.send()`, no BTCMesh chunking/ARQ involved)
     showed **even the short message failing to arrive** in that test run -
     contradicting the clean size-based explanation.
- Given the inconsistent, contradictory results across repeated live tests
  (the same class of message succeeding in one run and failing outright in
  another), this looks like general RF/hardware unreliability specific to
  this pair of devices, rather than an isolatable software bug:
  - Possible marginal RF link / antenna quality on one or both devices
  - Possible mutual interference from the two radios being physically very
    close together (receiver desensitization from a nearby active
    transmitter is a known real-world LoRa phenomenon)
  - Possible timing/settling sensitivity shortly after `connect()`, similar
    to the flakiness already seen and documented in Issue 12
- Not pursued further as a code fix since the signal itself isn't
  reproducible/consistent enough to isolate. If this resurfaces, worth
  re-testing with a different pair of physical devices to determine whether
  it's specific to this exact hardware pair, or trying a longer physical
  separation between the two devices to rule out mutual interference.
- **Retested 2026-07-19** while closing out Story 23.2's deferred real-hardware
  verification (both devices freshly power-cycled). Real connections (Meshtastic
  serial handshake + Bitcoin RPC over Tor) succeeded cleanly on both devices this
  time - the earlier "wedged, needs a power cycle" state was transient, not
  reproduced. Sent a real transaction (15 chunks) in both directions:
  - ESP32→Seeed (client `!3364d5a8` → server `!aee5ab3c`): failed exactly as
    before - **zero** chunks ever reached the server across all 4 attempts of
    chunk 1.
  - Seeed→ESP32 (client `!aee5ab3c` → server `!3364d5a8`): every chunk the
    client sent was received by the server **on its first arrival** (confirmed
    from the server's own logs, including chunks the client itself had to
    retry) - it was the **ACK replies travelling back to the client** that were
    being lost, not the chunks. This caused repeated client-side retries of
    chunks the server already had (correctly deduplicated and re-ACKed each
    time), and the transaction ultimately failed when chunk 3's retries were
    exhausted (4 attempts) purely from ACK loss, even though the server had
    held that chunk correctly since its first attempt.
  - This narrows the hypothesis: in the Seeed→ESP32 direction specifically,
    the forward (chunk) path is solid but the return (ACK) path is lossy
    enough to exhaust the current retry budget. Combined with the
    ESP32→Seeed direction still failing completely, this continues to look
    like an asymmetric RF/hardware issue tied to this specific device pair
    rather than anything in `server/receiver.py` or `client/sender.py` -
    confirmed by watching both `TransactionReceiver`'s duplicate-chunk
    handling and the client's stop-and-wait retry logic behave exactly as
    designed throughout.

## Protocol: Client retries after server has already completed the session

**Issue 17: If the final chunk's ACK or the broadcast ACK/NACK is lost in
transit, the client retries the last chunk - but the server has already
deleted the (now-completed) session, so the retry starts a confusing new
one** 🔍 OPEN — protocol-level gap, not yet fixed

- Reported by the user while testing: "if all chunks do get properly
  received, reassembled and broadcasted on server side, [and] the ACK or
  NACK after receiving all chunks is not received by the client, the client
  will retry sending the last chunk. While the server side already finished
  the session. So triggering a new session start at server and client never
  knowing it did actually get received, reassembled and/or broadcast
  already."
- Traced through `core/reassembler.py`'s `add_chunk()`: on successful
  reassembly (last chunk received), the session is deleted immediately
  (`del sender_sessions[tx_session_id]`) before the RPC broadcast even
  happens. If the chunk-level `BTC_CHUNK_ACK|...|ALL_CHUNKS_RECEIVED` or the
  final `BTC_ACK`/`BTC_NACK` reply is lost on the way back to the client,
  the client's own retry (same `session_id`, same last `chunk_num`) arrives
  at a server that has no memory of that session anymore - `add_chunk()`
  treats it as chunk 1 of a *brand new* session (since the session key was
  deleted), waits for the other chunks that will never come (the client
  isn't resending those, since it already got their ACKs), and eventually
  times out via `cleanup_stale_sessions()`, sending the client a confusing
  NACK for a transaction that was, in fact, already successfully processed
  (possibly already broadcast to the network) minutes earlier.
- This is a real, structural gap in the protocol's idempotency: the server
  has no way to recognize "I've already completed session X for sender Y"
  and simply resend the cached final result, instead of either silently
  creating a bogus partial session or leaving the client in the dark about
  a transaction that actually succeeded.
- Pre-existing in the current production `btcmesh_server.py` too (not
  introduced by Story 23.1's `server/receiver.py` - the same
  session-deletion-on-completion behavior lives in `core/reassembler.py`,
  unchanged by that story).
- Not fixed yet - worth its own story. Likely direction: have the server
  retain a short-lived record of recently-completed sessions (session_id →
  final result: TXID or error) for some grace period after completion, so a
  retransmitted last chunk for an already-completed session gets the cached
  final ACK/NACK resent directly instead of spawning a new phantom session.
